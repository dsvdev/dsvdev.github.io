Go Slices

Полное техническое руководство по слайсам в Go. Внутреннее устройство, опасные подводные камни shared backing array, правильный append и copy — всё что нужно знать чтобы не выстрелить себе в ногу.

1. внутреннее устройство

Слайс в Go — это не массив. Это заголовок (slice header) из трёх полей: указатель на backing array, длина и ёмкость. На 64-битной системе заголовок занимает ровно 24 байта.

Массив — фиксированный, тип-значение, хранится на стеке, размер известен на этапе компиляции. Слайс — динамический вид на часть массива. Заголовок передаётся по значению (копируется), но backing array — по указателю: оба слайса смотрят в одну и ту же память.

len — сколько элементов доступно прямо сейчас. cap — сколько памяти выделено начиная с ptr. Инвариант всегда: 0 ≤ len ≤ cap.

// Внутреннее представление — упрощённо (пакет reflect) type SliceHeader struct { Data uintptr // указатель на первый элемент backing array Len int // количество доступных элементов Cap int // ёмкость от Data до конца backing array } arr := [5]int{10, 20, 30, 40, 50} // массив — фиксированный, тип [5]int s := arr[1:4] // слайс: ptr→arr[1], len=3, cap=4 fmt.Println(s) // [20 30 40] fmt.Println(len(s)) // 3 fmt.Println(cap(s)) // 4 (до конца arr: позиции 1..4)

Когда слайс передаётся в функцию, копируется только заголовок (24 байта). Backing array не копируется. Поэтому изменения через s[i] = x видны снаружи. Но если функция делает s = append(s, ...), это меняет локальную копию заголовка — снаружи изменений нет. Именно поэтому append принято возвращать явно.

2. make vs литерал

Литерал — когда элементы известны при инициализации или нужен пустой слайс не-nil. make — когда нужен контроль над len и cap, или слайс нужен как буфер для последующих append.

var s []T создаёт nil-слайс. []T{} создаёт не-nil пустой слайс. Для большинства операций разницы нет, но для JSON-сериализации — принципиально важно.

// ── Литерал: элементы известны заранее ── primes := []int{2, 3, 5, 7, 11} // len=5, cap=5 empty := []string{} // len=0, cap=0, НЕ nil // ── nil-слайс: нулевое значение типа ── var ids []int64 // nil, len=0, cap=0, append работает // ── make: явный контроль len и cap ── buf := make([]byte, 0, 4096) // буфер: len=0, cap=4096 users := make([]User, 100) // 100 нулевых User, len=cap=100 matrix := make([][]float64, rows) // внешний слайс, len=cap=rows
make([]T, len) и make([]T, len, cap) различаются: в первом cap=len. Если потом будешь делать append — используй make([]T, 0, expectedSize), иначе первые len append будут писать за границу и расти сверх len.
3. append и рост capacity

append добавляет элементы в конец слайса. Если len < cap — элемент записывается на позицию s[len], len увеличивается, аллокации нет. Если len == cap — runtime выделяет новый backing array большего размера, копирует старые данные, и возвращает новый слайс.

Алгоритм роста в Go 1.18+: для маленьких слайсов (~<256 элементов) cap удваивается. Для больших — плавный рост по формуле newCap ≈ oldCap + (oldCap + 3·256) / 4. Точная формула может меняться между версиями — не полагайся на конкретные значения cap в логике программы.

s := []int{1, 2, 3} // len=3, cap=3 s = append(s, 4) // cap=6 (удвоился), len=4 — новый backing array s = append(s, 5, 6) // len=6, cap=6 — без аллокации s = append(s, 7) // cap=12, len=7 — снова удвоился // Вариадик: добавить все элементы другого слайса a := []int{1, 2} b := []int{3, 4, 5} a = append(a, b...) // a = [1 2 3 4 5] // Почему append возвращает новый слайс — наглядно func bad(s []int) { s = append(s, 99) // меняет локальную копию заголовка } // вызывающий не видит изменений! func good(s []int) []int { return append(s, 99) // явно возвращаем новый слайс }
append всегда нужно присваивать: s = append(s, v). Если пропустить присваивание, при росте backing array старая переменная указывает на устаревшие данные и имеет неверный len.
4. shared backing array

Когда создаёшь слайс из слайса (sub := s[1:3]), оба слайса разделяют один backing array. Изменение элемента через sub[0] меняет original[1]. Это самый частый источник неожиданных мутаций в Go.

Особенно коварно с append: пока у sub есть свободный cap, append пишет в ту же память что и original. Только когда cap переполнен — backing array расходятся и изменения перестают пересекаться.

original := []int{1, 2, 3, 4, 5} sub := original[1:3] // len=2, cap=4, ptr→original[1] sub[0] = 99 // ⚠ меняет original[1]! fmt.Println(original) // [1 99 3 4 5] // append в sub пишет в память original, пока есть cap sub = append(sub, 999) // len=3, cap=4 — пишет на original[3]! fmt.Println(original) // [1 99 3 999 5] // Как только cap переполнен — независимы sub = append(sub, 1, 2) // cap=4, нужно 5 → новый backing array sub[0] = 0 // original теперь не затронут
В конкурентных горутинах два слайса на одном backing array без синхронизации — гонка данных. go test -race поймает это только если гонка произошла в конкретном прогоне. Запускай тесты с -race в CI всегда.
// ── Защита 1: явная копия через copy ── safe := make([]int, len(sub)) copy(safe, sub) safe[0] = 0 // original не затронут // ── Защита 2: full slice expression [low:high:max] ── // Ограничивает cap, следующий append создаст новый backing array limited := original[1:3:3] // len=2, cap=2 (а не 4!) limited = append(limited, 99)// cap=2 → новый backing array, original цел fmt.Println(original) // [1 99 3 999 5] — без изменений
s[low:high:max] — full slice expression. max задаёт верхнюю границу cap, должен быть ≤ cap исходного слайса. Результат: len = high−low, cap = max−low. Используй при передаче подслайсов в функции, которые могут их расширять через append.
5. copy

copy(dst, src) копирует min(len(dst), len(src)) элементов и возвращает количество скопированных. Это единственный способ получить независимую копию данных слайса.

copy корректно работает для перекрывающихся срезов одного массива — стандартная библиотека гарантирует правильное направление копирования. Также copy принимает []byte и string как источник.

src := []int{1, 2, 3, 4, 5} // Полная копия dst := make([]int, len(src)) copy(dst, src) // dst = [1 2 3 4 5], независимый // Частичная: копируется min(len(dst), len(src)) small := make([]int, 3) n := copy(small, src) // n=3, small=[1 2 3] // copy для []byte ← string (без выделения строки в heap) b := make([]byte, 5) copy(b, "hello") // b = [104 101 108 108 111] // Сдвиг вправо на 1 (перекрытие — безопасно) s := []int{1, 2, 3, 4, 5} copy(s[1:], s) // s = [1 1 2 3 4]
Частая ошибка: dst := make([]T, 0); copy(dst, src) — скопируется 0 элементов, потому что len(dst)=0. Нужно: dst := make([]T, len(src)) или заранее правильный len.
6. nil slice vs empty slice

В большинстве операций nil slice и empty slice ведут себя одинаково: len и cap равны 0, append работает с обоими, range по обоим делает 0 итераций. Но есть два важных исключения: JSON-сериализация и reflect.DeepEqual.

nil slice сериализуется в null, empty slice — в []. Для большинства API-ответов ожидается пустой массив, а не null — это частый баг в Go-сервисах.

var nilSlice []int // nil, len=0, cap=0 emptySlice := []int{} // НЕ nil, len=0, cap=0 emptyMake := make([]int, 0) // НЕ nil, len=0, cap=0 fmt.Println(nilSlice == nil) // true fmt.Println(emptySlice == nil) // false // JSON: критическое различие data1, _ := json.Marshal(nilSlice) // null data2, _ := json.Marshal(emptySlice) // [] // reflect: тоже различаются eq := reflect.DeepEqual(nilSlice, emptySlice) // false // Операции: одинаково nilSlice = append(nilSlice, 1) // ok for range nilSlice {} // ok, 0 итераций
Правило для HTTP-хендлеров: если возвращаешь список и он пуст — возвращай []T{}, а не nil. Клиент получит [] в JSON вместо null. Исключение: если контракт явно требует null для «отсутствует» vs [] для «пустой список».
7. двумерные слайсы

Двумерный слайс в Go — это [][]T: слайс слайсов. Есть два подхода: независимые строки (N+1 аллокаций, строки не связаны) и один большой блок (2 аллокации, лучше кэш-локальность, но строки делят память).

const rows, cols = 3, 4 // ── Подход 1: независимые строки (N+1 аллокаций) ── matrix := make([][]int, rows) for i := range matrix { matrix[i] = make([]int, cols) // каждая строка независима } matrix[1][2] = 42 // ── Подход 2: один backing array (2 аллокации, лучше GC) ── storage := make([]int, rows*cols) grid := make([][]int, rows) for i := range grid { grid[i] = storage[i*cols : (i+1)*cols] } // grid[0][3] и grid[1][0] — соседние ячейки в памяти (кэш-дружелюбно)
В подходе 2 строки делят backing array. append в строку grid[0] за пределы cols перезапишет grid[1]. Защита: full slice expression при нарезке строк: storage[i*cols : (i+1)*cols : (i+1)*cols] — ограничивает cap строки.
8. производительность

Каждая реаллокация backing array — это аллокация в heap, копирование всех элементов, дополнительная нагрузка на GC. При n элементах без pre-allocation происходит ~log₂(n) реаллокаций. При n=1000 это уже ~10 ненужных копирований.

Если финальный размер известен (или оценим) — всегда передавай capacity в make. Это одна из самых дешёвых оптимизаций в Go.

// ❌ Медленно: O(log n) реаллокаций var result []string for _, item := range items { result = append(result, process(item)) } // ✅ Быстро: одна аллокация заранее result := make([]string, 0, len(items)) for _, item := range items { result = append(result, process(item)) } // ✅ Напрямую по индексу (размер точно известен) result := make([]string, len(items)) for i, item := range items { result[i] = process(item) } // Обнулить слайс без аллокации (переиспользование буфера) result = result[:0] // len=0, cap сохранён, память не освобождена
tip 1
go test -bench=. -benchmem показывает allocs/op. Хорошо оптимизированный горячий путь — 0 или 1 аллокаций. — benchstat сравнит до и после.
tip 2
sync.Pool + переиспользование через s[:0] убирает аллокации в обработчиках запросов. — Паттерн в fasthttp, zerolog, pgx.
tip 3
Конкатенация строк через strings.Builder — это тот же pre-allocated byte slice под капотом, но с удобным API. — bytes.Buffer — аналог для байт.
9. cheatsheet
операция способ примечание
nil-слайс var s []T nil, len=cap=0, append работает
пустой не-nil []T{} для JSON → [], не null
с capacity make([]T, 0, cap) одна аллокация, append без роста до cap
n нулевых элементов make([]T, n) len=cap=n, доступны сразу по индексу
добавить элементы s = append(s, v...) всегда присваивать, может расти
независимая копия dst := make([]T, len(s)); copy(dst, s) нет shared backing array
безопасная нарезка s[low:high:max] cap=max−low, защита от append в оригинал
удалить элемент i s = append(s[:i], s[i+1:]...) O(n), порядок сохраняется
удалить без порядка s[i] = s[len(s)-1]; s = s[:len(s)-1] O(1), меняет порядок
обнулить (сохранить cap) s = s[:0] переиспользование буфера без аллокации
проверить nil s == nil empty slice ≠ nil, важно для JSON / reflect
слить два слайса s = append(a, b...) a должен иметь достаточный cap или вырастет
