Архитектура
приложений

Слоистая и гексагональная архитектуры. Как структурировать код, чтобы он не превратился в монолитное болото через полгода.

01 — слоистая архитектура
Layered Architecture
Код разбивается на горизонтальные слои. Каждый слой зависит только от слоя ниже. Зависимости текут строго в одну сторону: сверху вниз.
01
Presentation
HTTP / gRPC / CLI
Принимает запросы снаружи. Валидирует входные данные, сериализует ответы. Не содержит бизнес-логики. Знает только о слое ниже — Service.
↓ зависит от
02
Business Logic
Service / Use Case
Сердце приложения. Оркестрирует операции, реализует правила домена. Не знает про HTTP, БД или очереди — только про данные и логику.
↓ зависит от
03
Data Access
Repository / DAO
Абстракция над хранилищем. Реализует CRUD-операции. Business Logic не знает как именно данные хранятся — SQL, Redis, файл.
↓ зависит от
04
Infrastructure
PostgreSQL / Redis / S3
Реальные технологии: база данных, брокер сообщений, файловое хранилище. Ничего не знает об остальных слоях. Просто хранит и отдаёт данные.
проблемы
Жёсткая связность
Business Logic напрямую зависит от Data Access слоя. Чтобы поменять PostgreSQL на MongoDB — нужно переписывать бизнес-логику.
Сложность тестирования
Unit-тесты для Service слоя требуют реальной (или фейковой) БД. Нет чистого способа подменить инфраструктуру без изменения кода.
Нарушение DIP
Высокоуровневые модули зависят от низкоуровневых. Это нарушение Dependency Inversion Principle из SOLID.
Утечка абстракций
SQL-модели часто проникают в бизнес-слой. Детали хранилища (транзакции, джойны) начинают влиять на логику.

02 — гексагональная архитектура
Hexagonal Architecture
Ядро домена изолировано от внешнего мира. Всё внешнее (HTTP, БД, Kafka) подключается через порты и адаптеры. Алистер Кокберн, 2005. Также известна как Ports & Adapters.
← Кликни на адаптер для описания
принципы
🔌
Порты
Интерфейсы, которые определяет ядро. Описывают что нужно домену — без деталей реализации.
🔧
Адаптеры
Конкретные реализации. HTTP-хендлер, SQL-репозиторий, Kafka-консьюмер. Реализуют порты.
🧪
Тестируемость
Ядро тестируется без БД и сети. Mock реализует порт — и готово. Никаких интеграционных зависимостей в unit-тестах.

domain/port.go
package domain

// Port — интерфейс, определённый ядром домена
// Ядро не знает ни про SQL, ни про HTTP

type User struct {
    ID    int64
    Email string
    Name  string
}

// UserRepository — порт (входящий)
type UserRepository interface {
    FindByID(id int64) (*User, error)
    Save(user *User) error
}

// UserService — ядро домена, зависит от порта
type UserService struct {
    repo UserRepository // интерфейс, не конкретный тип
}

func (s *UserService) GetUser(id int64) (*User, error) {
    user, err := s.repo.FindByID(id)
    if err != nil {
        return nil, fmt.Errorf("user not found: %w", err)
    }
    return user, nil
}
adapter/postgres/user.go
package postgres

// Адаптер: реализует порт domain.UserRepository
// Знает про SQL. Ядро домена — не знает.

type UserRepo struct {
    db *sql.DB
}

// Проверка соответствия интерфейсу на этапе компиляции
var _ domain.UserRepository = (*UserRepo)(nil)

func (r *UserRepo) FindByID(id int64) (*domain.User, error) {
    var u domain.User
    err := r.db.QueryRow(
        "SELECT id, email, name FROM users WHERE id = $1", id,
    ).Scan(&u.ID, &u.Email, &u.Name)
    if err != nil {
        return nil, err
    }
    return &u, nil
}

func (r *UserRepo) Save(u *domain.User) error {
    _, err := r.db.Exec(
        "INSERT INTO users (id, email, name) VALUES ($1, $2, $3)",
        u.ID, u.Email, u.Name,
    )
    return err
}

package domain_test

// Тест без БД. Mock реализует порт — и всё.

type mockUserRepo struct {
    users map[int64]*domain.User
}

func (m *mockUserRepo) FindByID(id int64) (*domain.User, error) {
    u, ok := m.users[id]
    if !ok {
        return nil, errors.New("not found")
    }
    return u, nil
}

func (m *mockUserRepo) Save(u *domain.User) error { return nil }

func TestGetUser(t *testing.T) {
    repo := &mockUserRepo{
        users: map[int64]*domain.User{
            1: {ID: 1, Email: "ghost@dev.io", Name: "Ghost"},
        },
    }
    svc := domain.NewUserService(repo)

    user, err := svc.GetUser(1)
    assert.NoError(t, err)
    assert.Equal(t, "ghost@dev.io", user.Email)
    // Никакой БД. Никакого Docker. Чисто.
}
